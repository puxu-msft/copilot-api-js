# GHC API 处理路径独立扫描（Codex 260330-1）

## 扫描基线

- 参考仓库：`refs/github-copilot-chat`
- 扫描时间：2026-03-30
- 本地扫描 commit：`6ad6a351c60c8dab1b9a1e620ef9156b28005893`

本文件是**独立扫描记录**，不依赖 `docs/2603-sync-ghc-api/` 现有分析结论，直接按源码调用链梳理 GHC 对 Copilot API 的处理路径。

## 总体调用链

GHC 的主链路可以概括为：

1. **鉴权与模型元数据**
   - `copilotTokenManager.ts` 负责拿 Copilot token
   - `modelMetadataFetcher.ts` 用 token 请求 `/models`
   - `endpointProviderImpl.ts` 用模型元数据实例化 `CopilotChatEndpoint`

2. **按模型选择协议与端点**
   - `chatEndpoint.ts` 读取 `supported_endpoints`
   - 决定走：
     - `/chat/completions`
     - `/responses`
     - `ws:/responses`
     - `/v1/messages`

3. **构造请求体**
   - OpenAI Responses：`responsesApi.ts`
   - Anthropic Messages：`messagesApi.ts`
   - 传统 Chat Completions：`networking.ts` + `openai.ts`

4. **发请求**
   - HTTP：`networking.ts`
   - WebSocket：`chatMLFetcher.ts` + `chatWebSocketManager.ts`

5. **流式响应归一化**
   - Responses：`responsesApi.ts`
   - Messages：`messagesApi.ts`
   - Chat Completions SSE：`chatStream.ts` / `stream.ts`

## 1. 模型元数据与路由决策

### 1.1 `/models` 元数据是核心输入，不是旁路信息

`endpointProvider.ts` 里的 `IModelAPIResponse` 已经把这些字段放进核心模型描述：

- `capabilities.supports.*`
- `billing`
- `warning_messages` / `info_messages`
- `supported_endpoints`

其中 `supported_endpoints` 枚举值明确包含：

- `/chat/completions`
- `/responses`
- `ws:/responses`
- `/v1/messages`

这不是展示字段，而是后续路由的直接依据。

### 1.2 模型刷新策略是“10 分钟 + 仅活跃会话”

`modelMetadataFetcher.ts` 的刷新机制：

- 无缓存时一定刷新
- 有缓存时超过 10 分钟才刷新
- VS Code 非活跃状态不刷新
- 请求 `/models` 被 429 时，如果已有缓存则直接保留旧数据并返回

这说明 GHC 把模型元数据当成**长期运行时状态**管理，而不是启动时一次性快照。

### 1.3 ChatEndpoint 直接按 `supported_endpoints` 选协议

`chatEndpoint.ts` 中的三个核心 getter：

- `useResponsesApi`
- `useWebSocketResponsesApi`
- `useMessagesApi`

对应含义是：

- 只要模型声明支持 `/responses`，就可能走 Responses
- 如果声明支持 `ws:/responses`，且实验开关允许，则可以走 WebSocket Responses
- Anthropic Messages 还要额外满足实验开关 `UseAnthropicMessagesApi`

所以 GHC 的真实设计不是“按 vendor 粗分”，而是：

- **vendor + capabilities + supported_endpoints + experiment flags** 联合决策

## 2. 请求头与基础网络层

### 2.1 通用请求头在 `networking.ts` 统一构造

`networking.ts` 会为 Chat 请求统一附加：

- `Authorization`
- `X-Request-Id`
- `OpenAI-Intent`
- `X-GitHub-Api-Version: 2025-05-01`
- `X-Interaction-Type`
- `X-Agent-Task-Id`

其中：

- `OpenAI-Intent` 用来表示请求来源场景
- `X-Interaction-Type` 细分主 agent / subagent / background
- `X-Agent-Task-Id` 直接复用 requestId

### 2.2 token / user-info / NES 等内部请求使用不同 API version

`copilotTokenManager.ts` 里 token 相关请求统一使用：

- `X-GitHub-Api-Version: 2025-04-01`

这说明 GHC 在头版本上不是“全局一个常量”，而是：

- Chat 主链路：`2025-05-01`
- token / auth / 内部接口：`2025-04-01`

### 2.3 网络错误重试是真正的“disconnectAll + retry once”

`networking.ts` 在 HTTP 请求失败时，如果命中这些错误码，会：

1. `fetcher.disconnectAll()`
2. 重新发一次相同请求

覆盖的错误码包括：

- `ECONNRESET`
- `ETIMEDOUT`
- `ERR_CONNECTION_RESET`
- `ERR_NETWORK_CHANGED`
- `ERR_HTTP2_INVALID_SESSION`
- `ERR_HTTP2_STREAM_CANCEL`
- `ERR_HTTP2_GOAWAY_SESSION`
- `ERR_HTTP2_PROTOCOL_ERROR`
- `ERR_FAILED`

这不是普通的“只要是 network_error 就 retry”，而是**显式枚举的单次重连策略**。

## 3. Anthropic Messages 路径

### 3.1 `messagesApi.ts` 是完整的请求构建器，不只是格式转换

它做的事情包括：

- 把 OpenAI/Raw 消息转成 Anthropic Messages 结构
- 生成 tool 列表
- 处理 tool search / defer_loading
- 计算 thinking config
- 生成 `output_config.effort`
- 生成 `context_management`
- 给 tools / system 自动补 `cache_control`
- 修补尾随 assistant message

换句话说，GHC 的 Messages 路径不是“透传为主”，而是**高度主动改写**。

### 3.2 Tool Search 分两种模式

`messagesApi.ts` + `anthropic.ts` 显示它有两套 tool search：

- **server-side**
  - 注入 `tool_search_tool_regex`
  - `type = tool_search_tool_regex_20251119`
- **client-side custom tool search**
  - 使用自定义 `tool_search`
  - 结果需要转成 `tool_reference`

server-side tool search 只在这些模型前缀启用：

- `claude-sonnet-4.5`
- `claude-sonnet-4.6`
- `claude-opus-4.5`
- `claude-opus-4.6`

### 3.3 Tool deferral 是正式的一等机制

GHC 不是简单透传 `defer_loading`，而是自己算：

- tool search 开启
- 位置是 agent / messages proxy
- 不是 subagent
- 不在 `IToolDeferralService.isNonDeferredTool()` 白名单

满足时才打 `defer_loading: true`。

并且它会先把 tools 分成：

- non-deferred
- deferred

再按顺序拼接，以便后续 `cache_control` 命中最大稳定前缀。

### 3.4 Thinking 配置是自动生成的

Messages 路径里：

- 如果模型支持 adaptive thinking，且没被显式禁用，就发：
  - `thinking: { type: 'adaptive' }`
- 否则按 `minThinkingBudget` / `maxThinkingBudget` / `max_tokens` 计算：
  - `thinking: { type: 'enabled', budget_tokens }`

这里 reasoning effort 不是独立开关，而是 thinking 开启后的附加配置：

- `output_config: { effort }`

### 3.5 Context editing 是 request body + beta header 双通路

Anthropic context editing 不是单一字段，而是两层：

1. `chatEndpoint.ts`
   - 加 `anthropic-beta: context-management-2025-06-27`
2. `messagesApi.ts`
   - 真正把 `context_management` 放进 body

并且 `getContextManagementFromConfig()` 会结合配置生成：

- `clear_tool_uses_20250919`
- `clear_thinking_20251015`

### 3.6 cache_control 自动注入是真实存在的，并且优先级很明确

`addToolsAndSystemCacheControl()` 的策略是：

1. 先统计 messages + system 里已有多少 `cache_control`
2. 最多只允许 4 个 breakpoint
3. 优先给**最后一个非 deferred tool**
4. 如仍有空位，再给**最后一个 system block**

这说明 GHC 对 prompt cache 的理解非常具体，不是“有就加”。

### 3.7 GHC 会主动修正常见 Anthropic 400 场景

至少包括：

- **尾随 assistant 消息**
  - 自动追加 `Please continue.`
- **tool_result 内容过滤**
  - 只保留 `text` / `image` / `document`
  - 空文本会被丢弃
- **图片 URL / base64**
  - 转成 Anthropic `image.source`
- **PDF**
  - 转成 `document` block
- **thinking round-trip**
  - `thinking`
  - `redacted_thinking`

这些都说明 GHC 把 Messages API 当成“需要主动适配的上游协议”，不是被动代理。

### 3.8 Messages 响应里还会回传 context management / server tool 数据

流处理器 `AnthropicMessagesProcessor` 会从 stream 中提取：

- `context_management`
- `server_tool_use`
- `tool_search_tool_result`
- thinking blocks
- 使用量里的 `server_tool_use.tool_search_requests`

所以 GHC 的 Anthropic 路径不仅处理请求，也完整消费了服务端扩展流事件。

## 4. OpenAI Responses 路径

### 4.1 `responsesApi.ts` 默认就把 Responses 当作“有状态协议”

请求构建时做的事情包括：

- 把历史消息转换成 `input`
- 从历史里提取 `previous_response_id`
- 提取 compaction data
- 提取 thinking data
- 转换 function call / function_call_output

也就是说，GHC 的 Responses 实现不是单次请求视角，而是**跨轮状态视角**。

### 4.2 Context management / compaction 是正式能力

如果模型不在 `modelsWithoutResponsesContextManagement` 黑名单里，并且配置启用：

- 自动注入
  - `context_management: [{ type: 'compaction', compact_threshold: 90% max prompt }]`

返回流里如果遇到 compaction output item，又会回传：

- `contextManagement: { type: 'compaction', id, encrypted_content }`

### 4.3 `previous_response_id` 与本地 stateful marker 联动

Responses 请求构建会：

- 从历史消息提取 stateful marker
- 放入 `previous_response_id`

Responses 流完成时又会：

- 从 `response.completed` 取 `response.id`
- 回传为新的 `statefulMarker`

这就是它避免反复发送全量历史的关键机制。

### 4.4 prompt_cache_key / verbosity / reasoning 都是原生构建项

Responses 请求里，GHC 会按配置主动加入：

- `text.verbosity`
- `reasoning`
- `include = ['reasoning.encrypted_content']`
- `prompt_cache_key = conversationId:modelFamily`
- `truncation = auto | disabled`

其中 `include` 不是被动透传，而是默认主动加。

### 4.5 Responses 流处理器不仅转文本，还消费语义事件

`processResponseFromChatEndpoint()` 处理这些语义事件：

- `response.output_item.added`
- `response.function_call_arguments.delta`
- `response.output_item.done`
- `response.reasoning_summary_text.delta`
- `response.reasoning_summary_part.done`
- `response.completed`

并从中产出：

- tool call begin / delta / done
- compaction data
- thinking summary
- encrypted reasoning
- stateful marker
- phase data

这说明 GHC 对 Responses 的处理层级，比传统 chunk-to-text 代理高很多。

## 5. WebSocket Responses 路径

### 5.1 WebSocket 不是独立协议栈，而是 Responses 的持久连接变体

`chatEndpoint.ts` 中：

- 模型必须声明 `ws:/responses`
- 还要开实验开关 `ResponsesApiWebSocketEnabled`

满足时，`chatMLFetcher.ts` 会走 `_doFetchViaWebSocket()`。

### 5.2 WebSocket 连接按 `conversationId + turnId` 复用

`chatWebSocketManager.ts` 的复用粒度是：

- 同一 conversation
- 同一 turn

这意味着：

- 同一轮 tool call 往返可以复用连接
- turn 变化就主动关闭旧连接

### 5.3 WebSocket 路径仍然依赖 `previous_response_id`

连接内部会维护最近一次 `response.id` 作为 `_statefulMarker`。

发送新请求时会检查：

- `body.previous_response_id` 是否与连接内 stateful marker 匹配
- 是否带了 compaction data

如果状态不一致，GHC 会记录 telemetry，并在必要时回退。

### 5.4 WebSocket 失败会透明降级到 HTTP

`chatMLFetcher.ts` 有一整套 fallback 逻辑：

- WebSocket 请求失败
- 如果 HTTP fallback 成功
  - 计一次 consecutive fallback
  - 关闭当前 WS connection
- 连续多次 WS→HTTP fallback 后，临时禁用后续 WS 请求

这说明 WebSocket 在 GHC 中是**性能优化层**，不是唯一传输路径。

## 6. 这次独立扫描补出来的几个关键结论

相对现有 `docs/2603-sync-ghc-api/`，这次直接扫源码后，最重要的补充有这些：

1. `supported_endpoints` 在 GHC 里是**真实路由输入**，不是仅供展示的元数据。
2. Anthropic Messages 路径是**强改写型适配层**，而不是透传层。
3. Responses 路径天然围绕：
   - `previous_response_id`
   - compaction
   - encrypted reasoning
   - prompt cache
   来设计。
4. WebSocket Responses 已经是 GHC 的正式一条主路径，但它始终保留 HTTP fallback。
5. GHC 对网络问题的 retry 不是抽象 `network_error`，而是显式错误码 + `disconnectAll()`。
6. GHC 的“头、body、流”三层是联动设计的：
   - beta header
   - request body capability
   - stream semantic event
   缺一层都不算真正支持某项能力。
