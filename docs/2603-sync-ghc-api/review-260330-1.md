# sync-ghc-api 文档审阅 260330-1

## 审阅范围

复核 `docs/2603-sync-ghc-api/` 下全部 6 篇分析文档与 `README.md`，对照当前仓库实现判断：

- “本项目现状” 是否属实
- 优先级排序是否仍然成立
- 各文档之间是否自洽
- 哪些建议仍然值得保留

本轮是**文档审阅**，不是功能实现审阅；重点在于这批分析文档现在还能不能作为后续开发依据。

## 结论

这组文档里有不少**有价值的方向判断**，但当前版本已经不能直接当作实施清单使用。

主要问题不是“分析方向完全错误”，而是：

- 多处“当前未实现”的判断已经被代码现状推翻
- `README.md` 的优先级总览没有随正文校正
- 个别文档把“未实现”写成了“完全没有”，遗漏了当前已经存在的半实现或代理透传能力
- 参考基线在当前工作区不可复核，削弱了文档可追溯性

如果要继续使用这组文档，建议先做一次**基于当前代码状态的重写/收敛**，而不是只追加零散修订。

## Findings

### 1. [高] `request-headers.md` 把已实现的请求头写成“缺失”，导致 P0 优先级失真

`docs/2603-sync-ghc-api/request-headers.md:10-17`、`22-23`、`27-68` 明确写到当前项目缺少：

- `X-GitHub-Api-Version`
- `X-Request-Id`
- `OpenAI-Intent`

但这些头已经在 `src/lib/copilot-api.ts:46-58` 中生成：

```ts
"openai-intent": opts?.intent ?? "conversation-panel",
"x-github-api-version": COPILOT_API_VERSION,
"x-request-id": randomUUID(),
```

因此，这篇文档当前只剩下两个可能还值得讨论的 gap：

- `X-Interaction-Type`
- `X-Agent-Task-Id`

文档现在把“补齐全部请求头”列为 P0，会直接误导后续工作量评估，因为其中大半其实已经完成。`README.md:24` 也因此被连带污染。

建议修正为：

- 把 `X-GitHub-Api-Version` / `X-Request-Id` / `OpenAI-Intent` 从缺口列表移除
- 单独评估 `X-Interaction-Type` / `X-Agent-Task-Id` 是否真的对当前代理有收益
- 说明本项目还额外维护了 `X-Interaction-Id` 会话级追踪头，避免重复设计

### 2. [高] `model-capabilities.md` 错误描述了当前路由机制，`supported_endpoints` 其实已经是核心决策输入

`docs/2603-sync-ghc-api/model-capabilities.md:77-81` 写道当前项目“通过 vendor 硬编码路由逻辑”，并把 `supported_endpoints` 描述成未来可保留的辅助参考。

这与当前实现不符：

- `src/lib/models/client.ts:56-72` 已经缓存 `supported_endpoints`
- `src/lib/models/endpoint.ts:23-58` 已将其封装为 `getEffectiveEndpoints()` / `isEndpointSupported()` / `isResponsesSupported()`
- `src/routes/chat-completions/handler.ts:111-132` 已基于端点支持情况在 `/chat/completions` 与 `/responses` 之间做路由决策
- `src/routes/responses/handler.ts:49-54` 与 `src/lib/anthropic/sse.ts:31-40` 也都在用端点支持做校验

也就是说，这不是“未来增强点”，而是**当前架构事实**。

这条误判会带来两个后果：

- 低估现有实现成熟度
- 错把“保存 supported_endpoints”列成 P1 工作项，而实际上该项已经完成

这部分建议应该改写为：

- `supported_endpoints`：已落地，文档应转为描述当前路由规则
- 真正未完成的项只剩模型列表定期刷新、thinking budget min/max 校验等

### 3. [高] `messages-api.md` 对 tool search / tool deferral 的现状判断已经过时，而且遗漏了真正的剩余缺口

`docs/2603-sync-ghc-api/messages-api.md:61-73` 说当前项目“未在请求构建中注入 tool search 工具”；`92-102` 说 `defer_loading`“未实现”。

这两条都已过时。当前实现中：

- `src/lib/anthropic/message-tools.ts:157-164` 会自动注入 `tool_search_tool_regex`
- `src/lib/anthropic/message-tools.ts:166-180` 会按规则设置 `defer_loading`
- `src/routes/messages/handler.ts:137-143` 已把 `preprocessTools()` 放到 sanitize 前执行，说明这条链路已接入主流程

因此，真正还值得讨论的问题不是“有没有实现”，而是：

- 当前 `modelSupportsToolSearch()` 仍只覆盖 Opus 4.5/4.6，是否需要扩到 Sonnet 4.5/4.6
- 当前 defer 规则是否与 GHC 的 non-deferred tool 集保持一致

此外，`messages-api.md:140-146` 把 tool result 过滤描述为“完全未做”，这也不准确。`src/lib/anthropic/sanitize/tool-blocks.ts:152-170` 已经在过滤非 `text` / `image` 的 user-side blocks。

但这里又暴露出文档漏掉的**真实问题**：当前过滤器并未保留 `document`。如果 Anthropic SDK 的 `ContentBlockParam` 中包含 document block，那么我们现在更接近“过滤逻辑不完整”，而不是“完全没过滤”。这比文档原结论更接近真实风险。

### 4. [中] `README.md` 与 `thinking-system.md` 在 adaptive thinking 上互相打架，且 README 还把已完成项继续列为 P0

`docs/2603-sync-ghc-api/README.md:24-27` 把 “adaptive thinking 支持” 列为应尽快采纳的 P0，语义上更像“代理应主动改写请求以使用 adaptive thinking”。

但 `docs/2603-sync-ghc-api/thinking-system.md:36-41` 的正式建议却是：

- 作为代理应尊重客户端配置
- 最多加一条日志提示，不要自动改写成 adaptive

这两种建议不能同时成立。一个是“自动收敛到 GHC 行为”，一个是“严格透传，最多告警”。

同一份 `README.md:27` 还把 “Sonnet 4.6 context editing 支持” 列为 P0，但 `src/lib/anthropic/features.ts:45-55` 已经把 Sonnet 4 / 4.5 / 4.6 都纳入 `modelSupportsContextEditing()`。

所以这里至少要做两件事：

- 统一 adaptive thinking 的设计立场
- 从总览里移除已完成的 `modelSupportsContextEditing` 项

否则 `README.md` 作为总索引已经不可信。

### 5. [中] `responses-api.md` 与 `network-resilience.md` 普遍低估了现有 Responses 能力，尤其是 WebSocket transport

`docs/2603-sync-ghc-api/network-resilience.md:76-85` 说“本项目的 Responses API 使用 HTTP POST，WebSocket 仅用于 History UI”，并把 WebSocket transport 列为未来 P2。

这和当前仓库事实不符。`src/routes/responses/ws.ts:1-217` 已经实现了**客户端到代理**的 Responses WebSocket transport，支持 `response.create` 并把上游 SSE 转成 WS JSON 帧。

更准确的表述应该是：

- 代理↔上游：仍然是 HTTP/SSE
- 客户端↔代理：Responses WebSocket 已实现
- 未来可评估的是“是否做代理↔上游 WebSocket”，而不是“项目完全没有 WS transport”

同类问题也出现在 `docs/2603-sync-ghc-api/responses-api.md`：

- `76-84` 把 `previous_response_id` 描述成“未处理”，但 `src/types/api/openai-responses.ts:115-138` 已定义字段，`src/routes/responses/handler.ts:39-62` 与 `src/lib/openai/request-preparation.ts` 也没有剥离它，本质上已经透传
- `166-172` 把 verbosity 写成“未实现”，但 `text.verbosity` 同样已在类型中声明且通过原样透传可用
- `46-56` 对 `context_management` 的描述也更接近“未自动注入”，而不是“项目没有这项能力”

这些不是文字细节，而是会直接影响后续任务拆分：透传能力和主动注入能力不是一个量级。

### 6. [低] 文档引用的参考基线在当前工作区不可复核，降低了整组文档的可审计性

`docs/2603-sync-ghc-api/README.md:3-7` 把分析基线写成 `refs/vscode-copilot-chat/` 某个 commit，但当前工作区并不存在这个 checkout。

这意味着：

- 文档中的 GHC 代码片段无法在本仓库内复核
- “基准版本”声明对后续审阅者没有操作价值
- 一旦正文与当前实现冲突，只能依赖作者转述，不能直接回源验证

这不是正文 correctness 问题，但会显著降低文档可信度。至少应补一个可复核来源：

- 本地归档路径
- 提交哈希对应的仓库 URL
- 或把关键引用片段连同文件路径固化到文档中

## 仍然有效的建议

尽管这批文档有明显过时内容，下面这些方向仍然值得保留：

- `cache_control` 自动注入：当前仓库只声明了类型，未见自动注入链路；这仍是有价值的真实缺口
- thinking budget min/max 校验：当前 `src/lib/anthropic/request-preparation.ts` 只做 `< max_tokens` 裁剪，没有按模型元数据做上下界规范化
- `output_config` / `reasoning effort` 的可用性验证：当前确实还在 `COPILOT_REJECTED_FIELDS` 中剥离 `output_config`
- 模型列表定期刷新：`src/start.ts` 当前仍是在启动时 `cacheModels()` 一次，长期运行实例确实可能落后
- `X-Interaction-Type` / `X-Agent-Task-Id` 是否有必要补齐：可以作为独立小项重新评估，但不应再与已存在的请求头混写

## 建议的修订方式

不建议继续在现有 6 篇文档上做零散补丁。更稳妥的方式是：

1. 先重写 `README.md` 的优先级总览，只保留“当前还未完成”的事项。
2. 把每篇文档改成三段式：
   - GHC 做法
   - 当前仓库真实状态
   - 剩余差距
3. 明确区分三类能力：
   - 已实现
   - 已透传但未主动增强
   - 完全未实现
4. 对无法在当前工作区复核的 GHC 结论，加“待回源验证”标记。

## 总结

这组分析文档不能直接作为开发 backlog，因为其中多条“gap”已经不是 gap。

当前最需要修的是**文档与代码现状的对齐**，不是立刻照着 `README.md` 的 P0/P1 清单开工。完成一次基于现状的收敛后，这批材料仍然可以保留其参考价值，尤其是在 `cache_control`、thinking budget 校验、模型刷新策略这些真实未完成项上。
