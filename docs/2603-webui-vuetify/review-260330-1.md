## Findings

### 1. [中] `VHistoryPage` 仍把 cursor 分页写成 `v-pagination`，与当前数据模型不匹配

文档在 `VHistoryPage` 的重写方案中写了“分页控件用 `v-pagination`”（`docs/2603-webui-vuetify/README.md:379-385`）。

但当前 History store 不是页码分页，而是典型 cursor 分页：

- `ui/history-v3/src/composables/history-store/useHistoryData.ts:16-18` 暴露的是 `nextCursor` / `prevCursor` / `total`
- `ui/history-v3/src/composables/history-store/useHistoryData.ts:66-83` 获取列表时也是通过 `cursor` + `direction`
- `ui/history-v3/src/composables/history-store/useHistoryData.ts:172-179` 只有 `loadNext()` / `loadPrev()` 两个翻页动作
- 当前 `VHistoryPage` 也正是按这个模型渲染 “Newer / Older” 两个按钮（`ui/history-v3/src/pages/vuetify/VHistoryPage.vue:198-221`）

`v-pagination` 适合已知页码、总页数、当前页 index 的模型；文档没有给出把 cursor 分页转换成页码分页所需的额外状态和行为定义。因此这条如果直接照做，实施时很可能会走偏，要么引入伪页码，要么又回退成上一版按钮。

建议改成下面两种之一：

- 保持 cursor 模型，继续使用 `v-btn` / `v-btn-group` 实现 “Newer / Older”
- 如果坚持改成 `v-pagination`，先在文档里补清楚页码状态来源、总页数定义、游标与页码的映射策略

## Open Question

### 1. `useAppTheme()` 的 `useTheme()` 调用上下文仍需在实现时验证

文档已经主动标出了这一点：`useAppTheme()` 内部调用 Vuetify 的 `useTheme()`，而 `App.vue` 的 `<script setup>` 执行时机早于 `v-app` 模板渲染；如果 Vuetify 4 对上下文有严格要求，就需要把这段逻辑下沉到 `v-app` 内部 wrapper 组件（`docs/2603-webui-vuetify/README.md:268-301`）。

这不是当前文档错误，因为文档已经明确把它当作待验证事项写出来了。但它仍然是实施前需要先确认的技术前提，不能直接假设示例代码可无条件落地。
