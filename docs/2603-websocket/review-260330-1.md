## Findings

### 1. [高] `findReusable()` 的签名仍不足以表达文档自己定义的复用判定条件

当前 `connection-management.md` 一方面把 manager 接口定义为：

```ts
findReusable(previousResponseId: string): UpstreamWsConnection | undefined
```

见：

- `connection-management.md:11-12`

但另一方面又明确写出复用条件应是：

- `statefulMarker` 匹配
- 模型一致
- 非 busy

见：

- `connection-management.md:146-153`

这里的“非 busy”可以由 manager 内部检查已有连接状态来完成，但“模型一致”在当前签名下并没有足够输入。

`previous_response_id` 本身只能表达状态关联，不能表达：

- 当前请求的 `payload.model`
- `resolvedModel.id`
- 或者任何可用于复用分桶的 model identity

这意味着当前文档实际上存在一个残余矛盾：

- 文档要求 `findReusable()` 在匹配时校验“模型一致”
- 但接口却没有给它做这件事所需的输入

这个问题会直接影响实现设计：

- 要么 manager 偷偷依赖外部上下文，导致接口名义上纯净、实际上有隐藏依赖
- 要么实现阶段发现签名不够，再次回头改接口

建议：

1. 把 `findReusable()` 的输入显式升级为至少包含 marker + model identity，例如：

```ts
findReusable(opts: {
  previousResponseId: string
  model: string
}): UpstreamWsConnection | undefined
```

或：

```ts
findReusable(opts: {
  previousResponseId: string
  reuseKey: string
}): UpstreamWsConnection | undefined
```

2. 如果文档坚持“模型变化时不复用”，那就不要再保留只收 `previousResponseId` 的签名。
3. `implementation.md` 的连接管理器小节也应同步改成“按 marker + model bucket 查找”，而不是只写“查找匹配 `statefulMarker`”。

### 2. [中] `implementation.md` 仍未把第四轮接受的接口变化全文同步到实施步骤和伪代码

`review-260330-1-reply.md:126-150` 已经明确接受了第四轮的三类修正：

- `connect()` / `sendRequest()` 接收 signal
- `findReusable()` 需要考虑 busy / 模型变化
- Phase 3 owner 文件需要精确化

其中 owner 文件这一部分已经基本同步完成，但 implementation 正文里仍残留几处旧精度描述：

#### 残留 A：1.3 连接实现仍是旧接口摘要

`implementation.md:64-69` 现在仍写：

- `connect()` — 建立 WSS 连接
- `sendRequest(payload)` — 发送 `{ type: "response.create", ...payload }`

这里没有同步出：

- `connect(opts?: { signal?: AbortSignal })`
- `sendRequest(payload, opts?: { abortSignal?: AbortSignal })`

这会让实现者回到旧接口心智。

#### 残留 B：1.4 连接管理器仍写成“只按 marker 查找”

`implementation.md:73-78` 现在仍写：

- `findReusable(previousResponseId)` 遍历连接查找匹配 `statefulMarker`

但 `connection-management.md:146-153` 已经把规则提高到了：

- marker 匹配
- 模型一致
- 非 busy

也就是说，实施计划这里仍然比设计正文少了一层关键条件。

#### 残留 C：1.5 伪代码没有把 signal / 模型匹配同步进来

`implementation.md:84-111` 的伪代码当前仍是：

- `manager.findReusable(payload.previous_response_id)`
- `conn.connect()`
- `conn.sendRequest(payload)`

这里没有体现：

- `findReusable()` 所需的模型身份输入
- `connect()` / `sendRequest()` 的 signal 传递
- Phase 3 abort 如何沿着 `createResponses()` 进入上游 WS 路径

问题不在于伪代码不够“可运行”，而在于它会直接影响实现者脑中的第一版接口。

建议：

1. 把 `implementation.md:64-69` 改成和 `connection-management.md` 一致的接口摘要。
2. 把 `implementation.md:73-78` 改成“按 marker + model identity + !busy 查找复用连接”。
3. 把伪代码至少同步到这个级别：

```ts
const reusable = payload.previous_response_id
  ? manager.findReusable({
      previousResponseId: payload.previous_response_id,
      model: payload.model,
    })
  : undefined

const abortSignal = /* shutdown + client abort composition */
if (!conn.isOpen) await conn.connect({ signal: abortSignal })
const events = conn.sendRequest(payload, { abortSignal })
```

4. 如果 `createResponses()` 实际拿不到 client abort signal，也应在文档里明确 signal 来源，而不是只写“Phase 3 会传播 abort”。

## 可改进项目

除了上面的 findings，这版文档在实施前还建议补齐这些点：

1. 把“连接是否 open”和“连接是否可复用”再拆得更明确一些。
   当前 `isOpen` 注释写成“open 且非 busy”，实现上未必是最清晰的命名。更稳的做法通常是：
   - `isOpen` 表示底层 socket 存活
   - `isBusy` 表示当前正在处理请求
   - manager 负责组合这两个条件决定是否可复用

2. 给 `findReusable()` 增加一个更接近职责的名字备选评估。
   如果最终判定条件不仅是“查 marker”，而是“按复用键查可用连接”，像 `findReusableConnection()` 或 `findByReuseKey()` 会更贴近真实职责。

3. 在测试清单里补一条更精确的 manager API 测试。
   当前已经有“模型变化 → 新建”和“busy → 新建”，但最好再显式补一条：
   - `findReusable()` 在 marker 命中但 model 不同的情况下返回 `undefined`

4. 在 Phase 3 文档里补一句“复用现有 `raceIteratorNext()` 语义，而不是为 WS 单独发明 abort 机制”。
   这样实现会更贴近当前仓库的流控模式。
